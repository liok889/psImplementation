Methodology

  • Both implementations process the same pixels (a numeric fixture decoded once from the reference PNG) with identical parameters.
  • analysis() and synthesis() are timed separately by a dedicated harness (bench/bench_main.cpp / bench/bench_js.js) that calls each function directly — not the stock binary, which runs both plus image I/O.
  • Each timed region includes that stage's own setup (filter construction, pyramid allocation, FFT plans), so the comparison is end-to-end per stage.
  • Reproduce with bash bench/run.sh.

Interpretation

  • The pure-JavaScript port runs ~3× slower than optimized native C++ — a strong result, given the native side uses FFTW's hand-tuned SIMD kernels while the JS uses a custom double-precision FFT with no SIMD.
  • Single-threaded is the fair comparison (JS is single-threaded) and the fastest native configuration here: enabling the reference's OpenMP + FFTW threads does not help at 256×256 — thread-spawn overhead dominates, and the multi-threaded binary is actually slower (~4.5 s wall for the full run vs. ~2.1 s single-threaded synthesis).
  • Numbers are from JavaScriptCore. A browser's V8 (Chrome) may differ somewhat, but the same single-threaded, no-SIMD constraints apply.
  • Cost scales as O(N log N) per FFT; both stages grow with image size, and synthesis additionally scales linearly with iteration count.
